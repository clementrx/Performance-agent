# Live multilingual evidence search — design

## 1. Problem

The evidence corpus is a static, packaged seed manifest (currently 10 studies,
targeting ~200) covering the MVP verticals (running, barbell strength). When an
athlete's goal falls outside that coverage — e.g. javelin throw — `search_evidence`
finds nothing relevant, and the coach can only say so honestly; it cannot go find the
actual literature.

The original product vision (confirmed by the project owner, distinct from what the
existing blueprint at `docs/superpowers/specs/2026-07-10-performanceagent-architecture-design.md`
§5 currently specifies) is: when an athlete states a goal, the agent should perform a
deep, multilingual literature search to find the best available training methods for
that specific goal — not be limited to whatever is already in the packaged corpus.

This spec extends the evidence system with a live, multilingual, verified search path,
without weakening the project's core anti-fabrication guarantee: **citations are only
ever rendered from corpus entries that have been validated (DOI/PMID resolved against
Crossref/PubMed) and grade-ceiling-checked** — the same guarantee that applies to the
packaged corpus today.

## 2. Non-goals

- No change to the packaged corpus (`seed_corpus.yaml`) or its maintainer verification
  workflow (`evidence/verify.py`) — those remain the human-curated, versioned-release
  path described in the existing blueprint.
- No general-purpose unverified web citation — anything without a resolvable DOI/PMID
  never becomes citable, full stop.
- No translation service dependency — language query terms are generated by the host
  agent (LLM), not by a translation API. The server only ever receives already-written
  query strings per language.

## 3. New MCP tools (`server/evidence_tools.py`)

### `search_evidence_live(topic: str, language_terms: dict[str, str]) -> LiveSearchResults`

Server-side fan-out, one tool call from the agent's perspective:

- `language_terms` maps an ISO language code to a search-term string the agent already
  translated for that language (e.g. `{"en": "javelin throw training periodization",
  "fr": "entraînement lancer javelot périodisation", "de": "Speerwurf
  Trainingsperiodisierung"}`). The agent decides which languages to include; the skill
  layer instructs a default set (en, fr, es, de, ru, no, sv, it, zh).
- For each `(language, term)` pair, the tool queries three keyless structured APIs:
  - **PubMed** E-utilities `esearch` + `esummary`/`efetch` (for `PublicationTypeList`
    and abstract).
  - **Crossref** `/works` search endpoint.
  - **Semantic Scholar** Graph API `/paper/search` (anonymous tier).
- Results are deduplicated across languages/sources by DOI (case-folded) or PMID —
  reusing the trimming logic already in `evidence/citations.py::_is_known_doi`.
- Every surviving candidate is verified the same way `evidence/verify.py` verifies
  packaged entries today (DOI resolves via Crossref, or PMID resolves via PubMed, title
  present). Candidates that don't resolve are dropped, not returned.
- Each verified candidate carries `suggested_study_type: StudyType | None` — populated
  directly from PubMed's `PublicationTypeList` when it unambiguously maps to a
  `StudyType` (e.g. `"Randomized Controlled Trial"` → `StudyType.RCT`,
  `"Meta-Analysis"` → `StudyType.META_ANALYSIS`, `"Systematic Review"` →
  `StudyType.SYSTEMATIC_REVIEW`). `None` when the source is Crossref/Semantic-Scholar-
  only or PubMed's tags don't map cleanly — in that case the agent must classify from
  the abstract, and `save_evidence`'s existing grading-ceiling validator still caps
  what it can claim.
- A source/language pair that times out or errors is skipped, not fatal — the tool
  returns whatever succeeded plus a `failed_sources: list[str]` field naming what
  didn't respond, so the agent can mention degraded coverage instead of silently
  under-searching.

```python
class LiveCandidate(TypedDict):
    title: str
    authors: list[str]
    year: int
    journal: str | None
    abstract: str | None
    doi: str | None
    pmid: str | None
    suggested_study_type: str | None  # StudyType value, or None
    source: str  # "pubmed" | "crossref" | "semantic_scholar"
    found_via_language: str

class LiveSearchResults(TypedDict):
    candidates: list[LiveCandidate]
    failed_sources: list[str]
```

### `save_evidence(entry: EvidenceEntry) -> WrittenFile`

- Validates `entry` through the existing `EvidenceEntry` Pydantic model — the grading
  ceiling (`_enforce_grading_ceiling`) and locator requirement (`_require_locator`)
  apply unchanged, so an agent cannot save an entry graded above what its study design
  supports, nor without a DOI/PMID.
- Rejects (clear `ValueError`, actionable message) if `entry.id` collides with an id
  already in the packaged corpus or the athlete's personal corpus.
- Appends the entry to `evidence_extra.yaml` in the athlete's data directory
  (`resolve_athlete_dir()`), creating the file if absent.
- Invalidates the process-lifetime evidence index cache (`_index.cache_clear()`) so the
  new entry is immediately searchable via `search_evidence` in the same session —
  no server restart required.

### `verify_reference(doi: str | None = None, pmid: str | None = None) -> VerificationResult`

- Thin MCP wrapper around the existing `evidence/verify.py::verify_entry` resolution
  logic (refactored to accept a bare doi/pmid rather than a full `EvidenceEntry`).
- Used for the web-search fallback path (§5): before the agent can propose a
  `save_evidence` call for something it found via general web search, it must call this
  to prove the reference actually resolves and get the registry's title/abstract for
  classification.

## 4. Corpus loading and merge (`evidence/corpus.py`)

```python
def load_corpus() -> list[EvidenceEntry]:
    entries = parse_manifest(_packaged_manifest_text())
    extra_path = resolve_athlete_dir() / "evidence_extra.yaml"
    if extra_path.exists():
        entries += parse_manifest(extra_path.read_text(encoding="utf-8"))
    return entries
```

- `evidence_extra.yaml` lives in `PERFORMANCE_AGENT_HOME` (or whichever path
  `memory.paths.resolve_athlete_dir()` resolves to) — never inside the installed
  package, never touched by a `performance-agent` upgrade, never shared with the
  official repo corpus.
- Live-discovered entry ids are namespaced `live-<slug>` (slug derived from title) to
  make collisions with packaged ids structurally unlikely; `parse_manifest`'s existing
  duplicate-id check still catches the rest.
- `search_evidence`, `check_citations`, `render_report`, `get_citation` require **no
  code changes** — they all consume `load_corpus()` / `_index()` already, so once
  merged, the entire existing anti-fabrication pipeline (hard citation gate, grade
  stars, PDF render refusal) applies to live-discovered entries exactly as it does to
  packaged ones.

## 5. Skill changes

`goal-assessment`, `program-generation`, and `program-adaptation` gain
`search_evidence_live`, `save_evidence`, and (fallback path) `verify_reference` in
their `tools:` frontmatter.

New protocol step, inserted after the existing `search_evidence` call, run on every
goal evaluation (not only as a fallback — confirmed default):

1. Call `search_evidence` against the local corpus (fast, as today).
2. Call `search_evidence_live(topic, language_terms)` — the agent writes translated
   query terms for the default language set (en, fr, es, de, ru, no, sv, it, zh; more
   can be added by a skill revision without a server change).
3. For each returned candidate:
   - `suggested_study_type` present → use it as-is; do not re-classify upward.
   - `suggested_study_type` absent → read the abstract and propose a `study_type` and a
     1-2 sentence `conclusions` summary, under the same discipline
     `CONTRIBUTING.md` already imposes on human contributors: no figure absent from the
     abstract, conservative wording, grading ceiling is enforced server-side regardless
     of what's proposed.
   - Call `save_evidence` for each candidate worth keeping.
4. If neither step 1 nor step 2 surfaces anything relevant for a given language, fall
   back to the host agent's own web search (`WebSearch`/`WebFetch`) in that language,
   looking for federations, theses, conference proceedings. Any DOI/PMID surfaced this
   way must pass `verify_reference` before a `save_evidence` call is even attempted —
   nothing unverified is ever proposed for the corpus.
5. Proceed with assessment/program generation as today, now against the expanded
   corpus.

**Honesty when nothing is found anywhere:** the skill explicitly instructs the agent to
say so plainly — "deep search performed, no directly applicable study found for X, here
is the closest available literature" — rather than force-fit an unrelated citation.

## 6. Error handling and rate limits

- Every HTTP call (PubMed, Crossref, Semantic Scholar) uses the same short timeout
  pattern already in `evidence/verify.py` (20s) and a polite delay between requests to
  the same API (reusing `_POLITE_DELAY_S`).
- A failed source/language pair is recorded in `failed_sources`, not raised — one dead
  API must not blank out results from the other two.
- All three APIs are queried anonymously (no API key) — consistent with the existing
  "no API key required for any core function" principle; this remains true for this
  opt-in feature too.
- `save_evidence` reuses existing Pydantic validation errors verbatim — an
  agent-proposed entry that violates the grading ceiling or lacks a locator fails with
  the same actionable message a human contributor gets today.
- Id collision with the packaged corpus raises explicitly rather than silently
  shadowing a packaged entry.

## 7. Testing

- `tests/evidence/` gains coverage for:
  - `load_corpus()` merge behavior: packaged only, packaged + extra present, extra file
    absent, extra file empty.
  - `save_evidence`: accepts a valid entry and makes it immediately searchable; rejects
    grading-ceiling violation; rejects missing locator; rejects id collision (against
    packaged and against existing personal entries).
  - `search_evidence_live`: HTTP calls mocked (no real network in CI, consistent with
    the rest of the suite) — dedup across languages/sources by DOI/PMID, PubMed
    `PublicationTypeList` → `StudyType` mapping for the types that map cleanly, `None`
    for ambiguous types, one source failing does not blank the others.
  - `verify_reference`: resolves via Crossref/PubMed, fails cleanly on an unresolvable
    locator.
- `tests/skills/` — extend the existing harness that checks `tools:` frontmatter
  against skill-referenced tool names to cover the three updated skills.
